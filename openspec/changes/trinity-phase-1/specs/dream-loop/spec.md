## ADDED Requirements

### Requirement: Dream-Judge-Act feedback loop
The system SHALL implement a feedback loop where: (1) the Dreamer generates ideas, (2) the Judge evaluates/connects them, (3) the Executor acts on approved plans, and (4) execution outcomes feed back to influence future Judge decisions and Dreamer seeds.

#### Scenario: Full loop in awake mode
- **WHEN** a user request enters the awake pipeline
- **THEN** the Judge evaluates it, the Executor acts on the verdict, and the execution outcome is stored in state for future Judge reference

#### Scenario: Full loop in sleep mode
- **WHEN** the dream state is activated
- **THEN** the Dreamer generates ideas, the Judge-as-connector deepens them, this iterates until breakthrough or max iterations, and the breakthrough is delivered to the user via Executor

### Requirement: Sleep state initialization
When entering sleep mode, the system SHALL initialize dream state keys: `dream_seeds` (empty list), `dream_deepening` (empty string), `dream_depth` (0), and `breakthrough` (null).

#### Scenario: Clean state at sleep start
- **WHEN** the dream_state LoopAgent begins
- **THEN** the session state contains initialized dream keys with empty/zero values

### Requirement: Dream depth tracking
The Judge-as-collaborator SHALL increment the `dream_depth` state key by 1 after each cycle. Breakthrough detection SHALL require `dream_depth` >= 3 before `exit_loop` can be called.

#### Scenario: Depth incremented each cycle
- **WHEN** the Judge-collaborator completes a cycle
- **THEN** `dream_depth` is incremented by 1

#### Scenario: Early exit prevented
- **WHEN** the Judge-collaborator detects a potential breakthrough at dream_depth < 3
- **THEN** it does NOT call `exit_loop` but instead notes the potential and continues deepening

### Requirement: Breakthrough definition
A breakthrough SHALL be defined as an insight that reframes an existing problem or reveals a non-obvious structural connection between previously unrelated concepts — something that could not arise through direct, awake-mode reasoning.

#### Scenario: Valid breakthrough detected
- **WHEN** the Judge-collaborator identifies an idea that reframes a known problem after sufficient depth
- **THEN** it stores the insight in `breakthrough` state with a description of what was reframed and calls `exit_loop`

#### Scenario: Good idea but not a breakthrough
- **WHEN** the Judge-collaborator identifies a useful but non-reframing idea
- **THEN** it adds the idea to `dream_deepening` and lets the loop continue

### Requirement: Dream seeds accumulation across iterations
Dream seeds generated by the Dreamer in each iteration SHALL accumulate in the `dream_seeds` state list, allowing subsequent iterations to build on all previous seeds.

#### Scenario: Seeds accumulate across cycles
- **WHEN** the Dreamer runs in iteration N
- **THEN** it can read all seeds from iterations 1 through N-1 via `{ dream_seeds? }` in its instruction

### Requirement: append_to_state tool
The system SHALL provide an `append_to_state` ADK tool that appends a value to a list-type state key or sets a scalar state key. Both Dreamer and Judge agents use this tool.

#### Scenario: Append to list key
- **WHEN** an agent calls `append_to_state` with a list key and a value
- **THEN** the value is appended to the existing list in session state

#### Scenario: Set scalar key
- **WHEN** an agent calls `append_to_state` with a scalar key and a value
- **THEN** the state key is set to the provided value

### Requirement: exit_loop tool availability
The ADK `exit_loop` built-in tool SHALL be available only to the Judge-collaborator agent in sleep mode. No other agent SHALL have access to `exit_loop`.

#### Scenario: Only judge-collaborator can exit
- **WHEN** the dream_state LoopAgent is constructed
- **THEN** only the `judge_collaborator` sub-agent has `exit_loop` in its tools list
